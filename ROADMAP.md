# Simple Trader Roadmap

这个文件的目标不是讲“大而全”的愿景，而是回答三个很实际的问题：

1. 现在我们已经做到了哪一步
2. 接下来最合理的升级顺序是什么
3. 要走到“国内期货实盘 trader”还差哪些层

---

## 1. 当前状态

目前 `simple_trader` 已经有两条线：

### v1: 极简回测版

对应文件：

- `engine.py`
- `strategy.py`
- `run_demo.py`

能力：

- 单标的
- 单仓位
- 天级别 K 线
- 简单均线策略
- 简单买入 / 卖出
- 账户权益统计

用途：

- 让人先看懂量化系统最小闭环
- 理解“行情 -> 信号 -> 执行 -> 收益”

### v2: 期货仿真入门版

对应文件：

- `futures.py`
- `futures_strategy.py`
- `data.py`
- `run_futures_demo.py`

能力：

- 单一国内期货合约
- 天级别 K 线
- 多头 / 空头基础语义
- 合约乘数
- 保证金比例
- 多种手续费模型
- 基础风险控制
- CSV 行情读取
- 逐日解释日志
- 风控层独立模块
- 仓位管理层
- 按风险预算自动计算开仓手数

用途：

- 从“量化玩具回测”升级到“有期货味道的仿真 trader”

### v3: 多合约组合版

对应文件：

- `portfolio.py`
- `run_portfolio_demo.py`
- `sizing.py`
- `risk.py`

能力：

- 一个账户同时交易多个合约
- 每个合约独立 `ContractSpec`
- 组合级资金 / 权益 / 保证金统计
- 组合级风控
- 单品种风控
- 仓位管理与风险预算

用途：

- 作为后续演进的主线版本
- 未来优先在这条线继续做 paper trading / live broker 抽象

## 2. 当前设计原则

为了确保这个工程能长期做下去，我们先明确一些原则。

### 先易懂，再完整

优先保证：

- 结构清楚
- 能运行
- 容易解释

而不是一开始就追求：

- 接所有接口
- 全部交易规则一次补齐
- 上来就做实盘

### 一次只跨一层复杂度

不直接从：

- 教学回测

跳到：

- 实盘自动交易

而是按层升级：

1. 回测
2. 仿真撮合
3. 风控
4. 组合管理
5. 网关抽象
6. 实盘接入

### 永远保留可运行 demo

每一阶段都应该有一个可直接执行的入口脚本。

这样做的好处：

- 以后回来能快速恢复上下文
- 每次修改后都能立即验证
- 不会堆一堆“还没跑通”的半成品

## 3. 推荐升级路线

下面是建议的推进顺序。

### Phase 1: 把期货仿真做扎实

这是下一阶段的重点。

目标：

- 让 `simple_trader` 从“教学版期货回测”进化成“像样的期货仿真核心”

建议任务：

1. 支持反手开仓
   现在更偏“一次只持有一个方向，再单独平仓”。
   下一步要支持：
   - 持有多单时遇到做空信号，先平多，再开空
   - 持有空单时遇到做多信号，先平空，再开多

2. 支持更真实的持仓状态
   未来需要考虑：
   - 多笔持仓
   - 加仓 / 减仓
   - 分批平仓

3. 支持滑点
   目前默认按收盘价成交，过于理想化。
   需要加入：
   - 固定滑点
   - 按最小价格变动单位滑点

4. 支持更真实的手续费模型
   目前是“每手固定手续费”。
   后面要支持：
   - 按手收费
   - 按成交金额比例收费

5. 支持基础交易日志
   输出内容至少包括：
   - 信号
   - 成交
   - 当前持仓
   - 当日权益

### Phase 2: 支持多合约

目标：

- 从“单一合约 trader”升级成“多合约组合 trader”

建议任务：

1. 支持按 `symbol` 分组喂数据
2. 支持多个合约同时持仓
3. 支持不同合约不同 `ContractSpec`
4. 支持组合总权益统计
5. 支持按合约分别生成交易记录

这是一个关键分水岭。  
走到这里，系统才开始接近真正有实用价值的期货量化框架。

当前状态：

- 已完成基础版
- 下一步应补齐组合级仓位控制和更清晰的组合汇总视图

### Phase 3: 加风险控制

目标：

- 让系统具备最基础的“不要乱死”的能力

建议任务：

1. 仓位限制
   - 单品种最大手数
   - 单次最大开仓比例

2. 保证金限制
   - 最大保证金占用
   - 最低可用资金保护

3. 止损和止盈
   - 固定点数止损
   - 百分比止损
   - 移动止损

4. 回撤控制
   - 最大回撤超限后停止开新仓

5. 交易时段限制
   - 只允许在指定时段开仓 / 平仓

当前状态：

- 已完成基础风控骨架
- 已支持总保证金、单品种保证金、单品种手数限制
- 下一步应补充止损、回撤控制和更清晰的风控分层

### Phase 4: 数据层升级

目标：

- 让数据输入从 demo CSV 升级为可复用的数据层

建议任务：

1. 定义统一数据接口
   比如：
   - `load_bars_from_csv`
   - `load_bars_from_sqlite`
   - `load_bars_from_parquet`

2. 支持分钟级数据
   当前是天级别。
   后面可以扩到：
   - 5 分钟
   - 1 分钟

3. 支持交易日历
   国内期货交易时段和普通股票不一样，后面需要单独处理。

4. 支持主力合约映射
   这是国内期货很核心的事情。
   以后如果要做连续合约或换月策略，这一层很重要。

### Phase 5: Broker 抽象分层

目标：

- 把“回测执行”和“真实下单”彻底隔离

建议做成三层：

1. `BacktestBroker`
   - 用历史数据回放
   - 用于回测

2. `PaperBroker`
   - 用实时行情
   - 不真实下单
   - 只做模拟交易

3. `LiveBroker`
   - 用实时行情
   - 发真实订单

这是走向实盘前最重要的一步。  
只有这层抽象干净了，策略才能不和具体柜台绑死。

当前状态：

- 已完成第一层骨架
- 策略层现在已经开始依赖最小 broker 接口，而不是绑死在具体 backtest broker 实现上
- `brokers.py` 里已经落地了 `BacktestBroker / PaperBroker / LiveBroker` 三种角色
- `FuturesBroker` 和 `PortfolioBroker` 现在已经明确归类为 `BacktestBroker`
- `PaperBroker` 现在已经支持第一版 `paper matching`
- 下一步应继续把 `PaperBroker` 扩成真正的实时订单簿 / 挂单撮合器，并给 `LiveBroker` 接统一网关适配层

### Phase 6: 国内期货实盘接入

目标：

- 真正连接 CTP 或其他国内期货接口

建议任务：

1. 定义统一订单模型
   包括：
   - 开仓 / 平仓
   - 多 / 空
   - 手数
   - 限价 / 市价语义

2. 定义订单状态机
   包括：
   - 已提交
   - 部分成交
   - 全部成交
   - 已撤单
   - 拒单

3. 接入行情网关
   - 实时行情订阅
   - 行情回调

4. 接入交易网关
   - 登录
   - 报单
   - 撤单
   - 成交回报

5. 建立本地状态同步
   - 当前订单
   - 当前持仓
   - 当前资金
   - 断线重连恢复

6. 加入生产级日志和告警
   - 控制台日志
   - 文件日志
   - 错误告警

## 4. 不建议现在就做的事

为了控制复杂度，下面这些事不建议太早做。

### 不建议一开始就做全异步架构

异步在实盘里很常见，但过早引入会让代码可读性下降很多。  
应该等：

- 单线程回测逻辑稳定
- Broker 抽象清晰
- 订单状态模型稳定

之后再引入。

### 不建议一开始就接 Web 管理后台

Web 页面是锦上添花，不是交易核心。  
真正重要的是：

- 交易模型清楚
- 风控可靠
- 状态可恢复

### 不建议一开始就混做股票和期货

这两个方向虽然都叫量化，但交易语义差很多。  
当前项目路线明确偏向：

- 国内期货

先把期货走通，再考虑是否抽象出股票版本。

## 6. 下一步就做什么

如果继续推进，推荐的下一个具体任务是：

1. 把 broker 抽象再往前推进一层
   目标是继续把已经落地的三层 broker 角色做成可运行实现：
   - `BacktestBroker`
   - `PaperBroker`
   - `LiveBroker`

   当前进展：
   - `BacktestBroker` 已稳定
   - `PaperBroker` 已有第一版即时成交式 `paper matching`
   - `LiveBroker` 仍是连接层骨架

2. 给组合版增加更清晰的汇总输出
   比如：
   - 独立的权益曲线导出
   - 每个合约累计收益表

3. 扩展止损层
   在固定百分比止损之外，增加：
   - 固定点数止损
   - 移动止损

原因：

- 组合版已经成了主线
- 下一步应该让主线先变得更“可观察、可控制、可迁移”
- 这比继续堆策略细节更重要

---

## 7. 一句总结

`simple_trader` 的目标不是复刻原来的大仓库，  
而是用更清晰、更容易维护的方式，一步一步长成一个真正能接近国内期货实盘的 trader。
